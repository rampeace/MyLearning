Custom STL-Style Iterator Intuition
===================================

Goal
----
Understand how custom containers support STL-style iteration.

Key Idea
--------
STL algorithms do not care about container internals.

They only care whether the container exposes valid iterators.

Iterators usually support:
    ++it      // move next
    *it       // access current value
    it != end // compare

Different containers implement traversal differently internally.


===========================================================
1. MyVector
===========================================================

Vector stores elements contiguously in memory.

Memory:
    1000 -> 10
    1004 -> 20
    1008 -> 30

Iterator can simply be:
    int*

because raw pointers already naturally support:
    ++p
    *p
    p != end

Example:

class MyVector
{
private:
    int* data;
    int size;

public:
    MyVector(int s)
    {
        size = s;
        data = new int[size];
    }

    ~MyVector()
    {
        delete[] data;
    }

    int& operator[](int index)
    {
        return data[index];
    }

    int* begin()
    {
        return data;
    }

    int* end()
    {
        return data + size;
    }
};

#include <iostream>
#include <algorithm>

int main()
{
    MyVector nums(4);

    nums[0] = 40;
    nums[1] = 10;
    nums[2] = 30;
    nums[3] = 20;

    std::sort(nums.begin(), nums.end());

    for (int n : nums)
    {
        std::cout << n << std::endl;
    }
}

Key Insight:
-------------
For contiguous containers, raw pointers themselves are valid iterators.


===========================================================
2. MyList
===========================================================

Linked list nodes are NOT contiguous.

Memory may look like:

    5000 -> Node(10)
    9000 -> Node(20)
    2000 -> Node(30)

Iterator can no longer use raw pointer arithmetic.

Instead, iterator internally stores:
    Node* current

and overloads:
    operator++
    operator*
    operator!=

Example:

struct Node
{
    int value;
    Node* next;
};

class Iterator
{
private:
    Node* current;

public:
    Iterator(Node* node)
    {
        current = node;
    }

    int& operator*()
    {
        return current->value;
    }

    Iterator& operator++()
    {
        current = current->next;
        return *this;
    }

    bool operator!=(const Iterator& other)
    {
        return current != other.current;
    }
};

class MyList
{
private:
    Node* head;

public:
    Iterator begin()
    {
        return Iterator(head);
    }

    Iterator end()
    {
        return Iterator(nullptr);
    }
};

Traversal:

for (auto it = list.begin(); it != list.end(); ++it)
{
    cout << *it << endl;
}

Key Insight:
-------------
For linked lists, iterator movement becomes:
    current = current->next

instead of pointer arithmetic.

template<typename Iterator>
void Algo(Iterator begin, Iterator end)
{
    Iterator it = begin;

    while (it != end)
    {
        cout << *it << endl;

        ++it;
    }
}

===========================================================
Final Mental Model
===========================================================

Each data structure chooses its own natural traversal strategy.

Vector:
    pointer arithmetic

List:
    next-node traversal

Tree:
    tree navigation

Hash table:
    bucket traversal

But externally STL sees only:
    ++it
    *it
    it != end

This abstraction allows STL algorithms to work generically across completely different data structures.

C++ Template Instantiation and Code Generation
==============================================

Goal
----
Understand how C++ templates work internally through:
- type substitution
- template instantiation
- compile-time code generation


The Big Idea
------------
A C++ template is not real executable code yet.

It is more like:
    a blueprint
    a recipe
    a generic code pattern

Actual code is generated only when concrete types are used.


===========================================================
1. Basic Template Example
===========================================================

Template:

template<typename T>
void Print(T value)
{
    std::cout << value << std::endl;
}

Usage:

Print(10);
Print(3.14);
Print("hello");


===========================================================
2. What the Compiler Conceptually Does
===========================================================

The compiler performs:
    type substitution

and generates concrete versions.

Conceptually:

void Print(int value)
{
    std::cout << value << std::endl;
}

void Print(double value)
{
    std::cout << value << std::endl;
}

void Print(const char* value)
{
    std::cout << value << std::endl;
}

This process is called:
    template instantiation

    Duck Typing — Intuition and Mental Model
========================================

Goal
----
Understand the idea of duck typing and how it relates to:
- C++ templates
- STL iterators
- generic programming


The Big Idea
------------
Duck typing means:

    "If something behaves correctly,
     we don't care about its actual type."

The name comes from the famous phrase:

    "If it walks like a duck
     and quacks like a duck,
     it is a duck."


===========================================================
1. Behavior Matters More Than Type
===========================================================

Traditional type systems often ask:

    "What class does this object belong to?"

Duck typing instead asks:

    "Can this object perform the required operations?"


===========================================================
2. Real-World Analogy
===========================================================

Suppose we need something that:
- walks
- quacks

Duck typing philosophy says:

    We don't care whether it is:
    - a real duck
    - a robot duck
    - a cartoon duck

If it behaves like a duck,
we can use it.


===========================================================
3. C++ Template Example
===========================================================

Generic STL-style algorithm:

template<typename Iterator>
void Algo(Iterator begin, Iterator end)
{
    Iterator it = begin;

    while (it != end)
    {
        cout << *it << endl;

        ++it;
    }
}

The template does NOT ask:

    "Is Iterator derived from some base class?"

Instead it simply tries:

    ++it
    *it
    it != end

If these operations compile:
    algorithm works

If not:
    compilation fails
